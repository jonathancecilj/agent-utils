## üìã The Review Workflow

### 1. Analysis Phase
Read the current resume data (usually in `lib/data.ts`). Identify:
-   Generic bullet points (e.g., "Worked on Kubernetes").
-   Missing metrics.
-   Unbalanced descriptions (too much detail on low-level tasks, not enough on high-level impact).

### 2. Inquiry Phase
Before suggesting a rewrite, ask the user specific "Extracting Gold" questions:
-   "What happened to the system if you *didn't* do this work?"
-   "Who were the key stakeholders you had to convince to get this launched?"
-   "What was the total scale (users, traffic, data volume) this project handled?"

### 3. Transformation Phase
Generate 3 versions of a rewrite:
-   **Version A (The Precision Engineer)**: High focus on technical accuracy and specific tools.
-   **Version B (The Executive Leader)**: High focus on business impact, ROI, and organizational change.
-   **Version C (The Pure Boaster)**: Maximum intensity, designed to "wow" at first glance.
